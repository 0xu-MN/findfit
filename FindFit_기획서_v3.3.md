## 1. 사업 배경 및 문제 정의

### 1.1 현장에서 확인한 진짜 문제

> 💡 창업가 캠프 현장에서 가장 많이 들은 질문: **"어떤 아이템을 해야 할지 모르겠어요"**

| 문제 | 상황 | FindFit의 해결 |
|------|------|---------------|
| 아이템 모름 | 뭘 해야 할지 막막함 | Agent — 줄글 입력 하나로 트렌드 + 타사 레퍼런스 자동 제시 |
| 레퍼런스 못 찾음 | 경쟁사 찾기 귀찮고 해석 어려움 | Agent (얕게 무료) + 리포트 (깊게 유료) |
| 반응을 모름 | 지인 의견은 편향 | 실제 사람 블라인드 검증 + NDA 보안 |

### 1.2 SNS 대비 차별점

| SNS에 올렸을 때 | FindFit 사용 시 |
|---------------|---------------|
| "너무 좋아요!" — 관계 유지용 빈말 | 냉정한 전문가의 실제 구매 의향 데이터 |
| 본계정에 실패 전시 — 자기 PR에 독 | 철저한 프라이버시 보장 |
| 정성적 감상 | 정량적 스코어 — 계속/피봇/중단 근거 |

### 1.3 경쟁 지형

| 서비스 | 특징 | FindFit과의 차이 |
|--------|------|----------------|
| ValidatorAI (글로벌) | AI 기반 / 20만 명+ / 대화형 AI 멘토 | 실제 사람 검증 없음 |
| IDEAGACHA (국내) | 모두의 창업 대상 / 2026.4 출시 | 실제 사람 검증 없음 |
| 오픈서베이 / 틸리언 | 기업 타겟 대형 리서치 | 수십~수백만원 — 초기 창업자 불가 |

> ✅ FindFit의 해자: 실제 사람이 직접 평가한 PSF/PMF 데이터. AI가 절대 대체할 수 없는 것.

## 2. 서비스 정의

### 2.1 한 줄 정의

**창업 준비자가 아이템 탐색부터 실제 사람 검증까지 한 플랫폼에서 끝내는 서비스**

### 2.2 핵심 플로우

| 단계 | 이름 | 내용 |
|------|------|------|
| STEP 1 | 아이템 탐색 | Agent 대화 → 트렌드 + 타사 레퍼런스 자동 조회 → 검증 등록 연결 |
| STEP 2 | 실제 사람 검증 | 프로젝트 등록 → NDA 서명된 Reviewer 매칭 → 평가 |
| STEP 3 | AI 리포트 | 결과 해석 + 타사 레퍼런스 비교 + 다음 액션 추천 |

### 2.3 역할 정의

| 역할 | 명칭 | 정의 |
|------|------|------|
| 의뢰자 | Creator | PSF/PMF 검증이 필요한 창업 준비자 / 1인 창업자 |
| 평가자 | Reviewer | 피드백 제공 후 사례금을 받는 도메인 전문가 / 실무자 |

> **[v3.2] 역할 전환 정책**
>
> Creator와 Reviewer는 동일 계정에서 역할 전환이 가능하다 (정식 출시 기준).
> **베타 기간에는 역할 전환 UI를 비활성화한다.** 베타에서 두 역할 모두 참여하고 싶은 경우, 별도 계정으로 가입한다.
> 정식 출시 시 DB 마이그레이션 후 역할 전환 기능을 활성화한다.

## 3. 프로젝트 등록 플로우 — STEP 0 상세

### 3.1 전체 진입 경로 3가지

```
경로 1 — STEP 0 → 아이템 탐색부터
경로 2 — STEP 0 → 아이템 있어요
경로 3 — Agent 패널에서 직접 유입
```

### 3.2 경로 1 — 아이템 탐색부터 시작

```
[프로젝트 등록하기] 버튼 클릭
        ↓
STEP 0 모달 표시
"어떻게 시작할까요?"

[🔍 아이템 탐색부터 시작]  [✅ 아이템 있어요]
        ↓
Agent 화면으로 이동
(Agent가 먼저 말을 건다 — 자동 시작)
        ↓
Agent 대화
"어떤 분야에 관심 있으세요?
 몇 가지 여쭤볼게요 😊"
        ↓
카테고리 파악 → 트렌드 조회 → 타사 레퍼런스 수집
        ↓
아이템 구체화 완료 시 Agent가 자연스럽게 유도
"이 방향으로 실제 사람들의 반응을 확인해볼까요?
 검증 의뢰를 등록하면 72시간 안에 결과를 받을 수 있어요"

[지금 바로 등록하기 →]
        ↓
STEP 1 (기본 정보) 이동
Agent에서 파악한 카테고리/방향 자동 채워짐
```

### 3.3 경로 2 — 아이템 있어요

```
[프로젝트 등록하기] 버튼 클릭
        ↓
STEP 0 모달 표시
        ↓
[✅ 아이템 있어요] 클릭
        ↓
바로 STEP 1 (기본 정보) 이동
Agent 거치지 않음 — 기존 등록 플로우 그대로
```

### 3.4 경로 3 — Agent 패널에서 직접 유입

```
홈 대시보드 우측 Agent 패널에서
Creator가 자유롭게 대화 중
        ↓
아이템이 구체화되면
Agent가 자연스럽게
"이 방향으로 검증해볼까요?
 [프로젝트 등록하기 →]"
        ↓
클릭 시 STEP 1으로 이동
대화에서 파악한 카테고리/방향 자동 채워짐
```

### 3.5 경로별 비교

| 진입 경로 | Agent 거치는가 | STEP 1 자동 채움 | 대상 |
|---------|-------------|----------------|------|
| STEP 0 → 아이템 탐색 | 강제 진입 (자동 시작) | O (탐색 결과 반영) | 아이템 없는 사람 |
| STEP 0 → 아이템 있어요 | 거치지 않음 | X (직접 입력) | 아이템 있는 사람 |
| Agent 패널 → 등록 유도 | 이미 거침 | O (대화 내용 반영) | Agent 사용 중인 사람 |

## 5. FindFit Agent 구조

### 5.1 Agent의 핵심 역할과 차별점

> Agent는 "실제 사람 검증으로 데려오는 입구"다. 일반 AI 챗봇과 달리 **실시간 시장 데이터 + 구조화된 단계별 질문**으로 창업 전문 컨설턴트 경험을 제공한다.

| 구분 | 일반 AI 챗봇 | FindFit Agent |
|------|-------------|---------------|
| 데이터 | 학습 데이터 기반 일반론 | 실시간 구글 트렌드 + 메타 광고 + 네이버 데이터 |
| 질문 방식 | 자유 대화 | 구조화된 단계별 토스트 선택 |
| 결과 | 텍스트 분석으로 끝 | 실제 사람 검증으로 자연 연결 |
| 국내 특화 | 없음 | 네이버 데이터랩 + 국내 시장 특화 |

### 5.2 실시간 데이터 소스

| 데이터 소스 | 가져오는 정보 | API | 베타 상태 |
|-----------|------------|-----|---------|
| 네이버 데이터랩 | 국내 검색량 추이 / 카테고리별 인기 키워드 | 공식 API (무료) | ✅ 활성 |
| Google Trends | 글로벌 검색 관심도 / 급상승 키워드 | pytrends (비공식) | ⛔ 베타 비활성 |
| 메타 광고 라이브러리 | 이 카테고리 광고 집행 업체 수 / 광고 메시지 | Meta Ad Library API (공개) | ⛔ 베타 비활성 |
| 웹 검색 MCP | 유사 서비스 수집 / 타사 레퍼런스 | Claude MCP | ✅ 활성 |

> **[v3.2] 베타 데이터 소스 운영 정책**
>
> 베타에서는 **네이버 데이터랩 + 웹 검색 MCP** 2개 소스만 운영한다.
>
> - **Google Trends 비활성 이유:** pytrends(Python)는 Supabase Edge Function(Deno)에서 실행 불가. 정식 출시 전 별도 Python 마이크로서비스 배포 후 활성화한다.
> - **메타 광고 비활성 이유:** Meta Ad Library API는 앱 심사(3~7일)가 필요하고, 국내 광고 데이터 커버리지 검증이 필요하다. 지금 바로 신청을 시작한다.
>
> 두 소스가 비활성인 상태에서도 네이버 데이터랩(국내 검색 트렌드)만으로 핵심 시장 인사이트 제공이 가능하다. Agent 메시지에서 비활성 소스 섹션은 표시 생략한다.

**이게 차별점이에요**
```
ChatGPT: "건강기능식품 시장은 성장하고 있습니다" (일반론)

FindFit Agent:
"건강기능식품 구독 키워드 검색량이 최근 3개월 28% 증가했어요
 메타 광고에서 이 카테고리 광고 집행 중인 업체가 47개예요
 그 중 시니어 전용은 3개뿐이에요 → 공백 시장이에요"
```

### 5.3 4단계 구조화 Agent 플로우

**Phase 1 — 분야 파악 (토스트 선택)**
```
Agent: "어떤 분야에 관심 있으세요?"

┌──────────────────────────────┐
│ ☐ 앱 / 소프트웨어             │
│ ☐ 커머스 / 이커머스           │
│ ☐ 헬스 / 웰니스               │
│ ☐ 교육 / 에듀테크             │
│ ☐ 푸드 / F&B                 │
│ ☐ 핀테크 / 금융               │
│ ☐ 기타 (직접 입력)            │
└──────────────────────────────┘

Agent: "지금 어느 단계에 있으세요?"

┌──────────────────────────────┐
│ ○ 아이디어만 있어요            │
│ ○ 시장 조사 중이에요           │
│ ○ 만들고 있어요               │
│ ○ 이미 출시했어요             │
└──────────────────────────────┘
```

**Phase 2 — 실시간 데이터 수집 (자동, 백그라운드)**
```
선택 즉시 동시에 4개 API 호출 (베타: 2개만 활성)
→ 네이버 데이터랩 (검색 트렌드) ← 베타 활성
→ Google Trends (관심도 추이) ← 베타 비활성, 출시 후 활성
→ 메타 광고 라이브러리 (광고 집행 현황) ← 베타 비활성, 출시 후 활성
→ 웹 검색 MCP (유사 서비스) ← 베타 활성

로딩 중 메시지: "시장 데이터 찾는 중이에요... 잠깐만요 🔍"
```

**Phase 3 — 데이터 결과 + 심화 질문 (토스트 선택)**
```
Agent: "헬스/웰니스 시장 데이터예요 📊

검색 트렌드: 최근 3개월 +28% ↑
메타 광고: 집행 업체 47개
시니어 전용: 3개뿐 (공백 시장)

이 중 어떤 방향이 마음에 드세요?"

┌──────────────────────────────┐
│ ○ 시니어 전용 건강기능식품     │
│ ○ 2030 맞춤 비타민            │
│ ○ 가족 단위 구독 패키지        │
│ ○ 직접 입력할게요             │
└──────────────────────────────┘
```

**Phase 4 — 타사 레퍼런스 + 등록 유도**
```
Agent: "시니어 전용 방향으로 비슷한 서비스 찾았어요

┌─────────┬──────────────┬─────────┐
│ 필리     │ 맞춤 비타민  │ 고가    │
│ 마이비누 │ 정기배송     │ 저가    │
│ 헬스밀   │ 식단 구독    │ 중간    │
└─────────┴──────────────┴─────────┘

시니어 전용 경쟁사는 거의 없어요!
이 방향으로 실제 반응 확인해볼까요?
72시간 안에 전문가 피드백을 받을 수 있어요"

[지금 검증 등록하기 →]
```

### 5.4 타사 레퍼런스 — 두 가지 방식이 함께 작동

**방식 1 — FindFit이 자동으로 찾아줌 (기본)**
```
Creator가 아이템을 입력하기만 하면
FindFit Agent가 알아서 수집

→ 웹 검색 MCP로 유사 서비스 자동 탐색
→ 서비스명 / 타겟 / 가격대 / 강점 / 약점 자동 정리
→ Creator는 아무것도 안 해도 결과 확인 가능
```

**방식 2 — Creator가 직접 추가 (선택)**
```
Creator가 이미 알고 있는 경쟁사가 있으면
STEP 1에서 URL 직접 입력 (선택 사항)

→ "혹시 참고하고 싶은 서비스가 있으면 URL을 추가해주세요"
→ 입력한 URL을 AI가 분석해서 요약
→ 자동 수집 결과에 추가로 합쳐짐
```

**두 가지 합산 결과**
```
Agent가 자동으로 찾은 것 (최대 3개)
+
Creator가 직접 추가한 것 (선택, 개수 제한 없음)
= 최종 타사 레퍼런스 세트

→ Agent Phase 4에서 얕게 미리보기 (무료)
→ AI 리포트 심층에서 리뷰어 응답과 교차 분석 (유료 9,900원)
```

| 구분 | Agent Phase 4 (무료) | 리포트 심층 (유료 9,900원) |
|------|---------------------|--------------------------|
| 시점 | 등록 전 | 검증 완료 후 |
| 수집 방식 | 자동 수집 + Creator 추가 합산 | 자동 수집 + Creator 추가 합산 |
| 깊이 | 얕게 — 서비스명 + 한 줄 요약 | 깊게 — 리뷰어 응답과 교차 분석 |
| 목적 | 아이디어 구체화 + 등록 결심 | 경쟁사 대비 강점/약점 전략 인사이트 |

## 6. 제품 구조

### 6.1 MVP 범위

| 구분 | 항목 | 베타 상태 | 비고 |
|------|------|---------|------|
| ✅ 만든다 | Agent (줄글 → 트렌드/레퍼런스 → 검증 연결) | 활성 (네이버+MCP) | 핵심 입구 |
| ✅ 만든다 | STEP 0 모달 (탐색부터 / 아이템 있어요) | 활성 | 신규 |

## 15. [v3.2 신규] 베타/정식출시 기능 전환 정책

> 기획서와 기술 명세서는 **정식 출시 기준**으로 작성되어 있다.
> 베타 기간에는 코드 레벨의 Feature Flag로 일부 기능을 잠가 운영하고,
> 준비가 완료된 기능부터 순차적으로 활성화한다.
> **문서는 수정하지 않는다. 코드의 환경변수만 바꾼다.**

### 15.1 전환 원칙

```
1. 문서는 정식 출시 기준으로 고정 — 베타 예외를 문서에 기록하지 않는다
2. 코드에 Feature Flag 구조 구현 — 환경변수 하나로 기능 ON/OFF
3. 베타 잠금 기능은 코드가 이미 완성된 상태 — 플래그만 바꾸면 즉시 활성화
4. 출시 준비 완료 조건을 사전에 명확히 정의
```

## 16. [v3.3 신규] Reviewer 경험 설계

### 16.1 전체 화면 구조 (5개)

| 화면 | 경로 | 역할 | MVP 우선순위 |
|------|------|------|------------|
| 프로젝트 피드 | /evaluator/available | 참여 가능한 의뢰 탐색 + 지원 | ① 1순위 |
| 프로젝트 상세 | /evaluator/projects/[id] | 의뢰 내용 확인 + 지원 결정 | ① 1순위 |
| 평가 폼 | /evaluator/review/[matchId] | 3단계 평가 제출 | ① 1순위 |
| 내 참여 현황 | /evaluator/reviews | 진행 중/완료 목록 + 상태 배지 | ② 2순위 |
| 사례금 정산 | /evaluator/wallet | 수입 내역 + 출금 신청 | ③ 3순위 (베타 2주차) |

### 16.2 Reviewer 지원 → 평가 전체 플로우

```
[Reviewer] 피드(/evaluator/available)에서 프로젝트 카드 발견
        ↓
프로젝트 상세(/evaluator/projects/[id]) 확인
        ↓
[지원하기] 클릭 → 지원 폼 모달
  · 닉네임 (플랫폼 표시 이름 — Creator한테는 "리뷰어 A"로 보임)
  · 이메일 (수락 알림 수신용 — 운영자만 볼 수 있음)
  · 도메인 태그 (헬스케어 / 커머스 / B2B... 복수 선택)
  · 한 줄 소개 (선택 — 운영자 매칭 참고용)
  · NDA 동의 체크박스
        ↓
"지원이 완료됐어요. 검토 후 이메일로 안내드릴게요" 화면
project_matches.status = 'pending'

        ↓ [운영자 액션 — Admin 화면]

/admin/applications에서 지원자 목록 확인
→ [수락] → status='accepted' + Resend로 평가 링크 이메일 자동 발송
→ [거절] → status='dropped' + 안내 이메일 발송 (선택)

        ↓

[Reviewer] 수락 이메일 수신 → 평가 링크 클릭
→ /evaluator/review/[matchId] 접근 (수락 상태인 경우만 허용)
        ↓
평가 폼 3단계 진행 → 제출 완료
→ project_matches.status = 'completed'
→ completed_count +1 → 전원 완료 시 AI 리포트 자동 생성
```

### 16.3 프로젝트 피드 카드 구성 요소

### 16.5 평가 폼 — 3단계 구조

**Step 1 — 서비스 이해하기 (읽기 전용)**
```
Creator가 등록한 서비스 소개 텍스트 표시
랜딩페이지 링크 / 데모 영상 (등록된 경우)
                                  [다음 →]
```

**Step 2 — 질문 답변**
```
Q1 / Q2 / ... (Light: 최대 5개 / Standard: 최대 9개)

🔒 필수 질문: 삭제/수정 불가 잠금 표시
                                  [다음 →]
```

**Step 3 — 제출 확인**
```
☐ 본 서비스 정보 공개 시 NDA에 따라 법적 책임이 따름을 인지합니다

                         [평가 제출하기 ✓]
```

> 제출 완료 화면: "감사합니다! 사례금은 마감 후 48시간 이내 입금돼요 🎉"

### 16.6 내 참여 현황 — 상태 배지 4종

### 18.3 purchase_intent — 두 신호의 구분
