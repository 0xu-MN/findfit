# FindFit — 다음에 뭐부터 고쳐야 하는지 (실행 계획)

> 기준 커밋: `e841b2c`. `_docs/파이프라인_코드리뷰_검증및계획_20260720.md`(항목별 검증 결과)를 실행 순서로 재정리한 문서. "무엇을, 어떤 순서로, 왜 그 순서로"에 집중.

## 지금까지 완료된 것 (참고)
- 크리에이터 등록 → 리뷰어 매칭 → 리뷰 제출 → AI 리포트 파이프라인 실데이터 연결
- RLS 전면 적용 + 컬럼 제한 뷰(창작자/리뷰어 신원 비노출)
- 관리자 유저 관리(`/admin/evaluators`) + 프로젝트 검수 큐(`/admin/requests`) 화면
- 정산 로직 RLS 붕괴 수정 (`lib/distribution/execute.ts` → 서비스 롤 클라이언트)

## 왜 이 순서인가

세 그룹으로 나눴다. **1그룹**은 "지금 만들어진 기능이 실제로 끝까지 동작하는가"에 직결된 것 — 사용자가 직접 못 느끼면 소용없는 부분. **2그룹**은 "사용자 경험 완결성" — 있어야 자연스러운데 빠진 것. **3그룹**은 "운영/정합성" — 당장 급하진 않지만 방치하면 나중에 더 큰 빚이 되는 것.

## 1그룹 — 파이프라인이 "완결"되게 만들기 (이번 주)

### 1-1. 리포트 생성을 완료 즉시 확실하게 트리거
**지금 상태**: 마지막 리뷰어가 제출하면 브라우저에서 `POST /api/ai-report/[id]`를 호출하는데, 이 요청이 실패(탭 닫힘/네트워크 끊김)하면 아무도 모르게 리포트가 안 생긴다. 다행히 크리에이터가 `/builder/reports/[id]`를 열면 그때 자동 생성되는 fallback이 있어서 "영영 안 생기는" 건 아니지만, "완료 즉시"는 보장 안 된다.

**할 일**:
1. `app/evaluator/review/[id]/page.tsx`의 `handleSubmit` 로직(답변 insert + 매칭 상태 갱신 + RPC 호출 + 완료 체크 + 리포트 트리거)을 새 서버 API 라우트 하나로 옮기기(예: `app/api/reviews/[matchId]/submit/route.ts`)
2. 이 라우트 안에서 `increment_completed_count` RPC 결과를 서버가 직접 확인하고, 도달했으면 그 자리에서 `generateAndSaveReport()`를 동기 호출(또는 최소한 서버 간 fetch로 트리거)
3. 브라우저는 이 라우트 하나만 호출하면 되므로 왕복 횟수도 줄어듦

**왜 1순위**: 이게 안 되면 "완료됐는데 리포트가 왜 안 나오죠?"라는 문의가 실제 서비스에서 제일 먼저, 제일 자주 나올 문제.

### 1-2. 리포트 생성 후 프로젝트 상태 정리
**할 일**: `lib/ai/generateReport.ts`의 `generateAndSaveReport()` 끝에 `projects.status = 'completed'` 업데이트 추가
**왜**: 1-1 작업하는 김에 같이 처리하는 게 효율적 (같은 함수 근처). 지금은 관리자 대시보드의 "진행중 프로젝트" 카운트가 실제로 끝난 프로젝트까지 세고 있음.

### 1-3. 계좌 등록 페이지 연결
**지금 상태**: `/reviewer/account-setup`(계좌 정보 입력) 자체는 다 만들어져 있는데 어디서도 링크가 없어서 리뷰어가 도달할 방법이 없다. 정산 로직은 방금 고쳤지만(1번 항목), 애초에 계좌를 등록한 리뷰어가 한 명도 없으면 정산이 실행돼도 전원 "계좌 미등록" 처리로 빠진다.
**할 일**: `app/evaluator/wallet/page.tsx` 또는 `components/shared/DashboardLayout.tsx`의 리뷰어 내비게이션에 "계좌 등록" 메뉴/배너 추가
**왜**: 방금 고친 정산 로직이 실제로 쓸모 있으려면 이게 선행돼야 함.

## 2그룹 — 사용자 경험 완결성 (2주 내)

### 2-1. 리뷰어 프로필 편집 화면
**지금 상태**: `app/evaluator/profile/page.tsx`가 빈 스텁. `domain_tags`/`level`은 매칭 점수 계산(`computeMatchScore`)에 실제로 쓰이는데 리뷰어가 설정할 방법이 없어서 전원 매칭 점수가 낮게(또는 0으로) 고정됨.
**할 일**: 이 화면에 `reviewer_profiles.domain_tags`(다중 선택 태그) 편집 UI 구현. `level`은 관리자가 부여하는 값이라 사용자가 직접 수정하는 항목은 아님(읽기 전용 표시만).

### 2-2. 리뷰어 매칭 자진 취소
**지금 상태**: 수락된 매칭을 리뷰어가 스스로 그만두는 방법이 없음. 관리자가 수동으로 거절 처리하는 것만 가능.
**할 일**: `/evaluator/reviews`(내 리뷰 목록) 화면에 "참여 취소" 버튼 + `project_matches.status='dropped'`로 바꾸는 API(`app/api/evaluator/matches/[id]/drop/route.ts` 신규, RLS상 `reviewer_id=auth.uid()`라 본인 매칭은 이미 UPDATE 가능하므로 서비스 롤 없이도 구현 가능)

### 2-3. 마법사 유실 필드 보존
**지금 상태**: 크리에이터가 마법사에서 입력하는 `occupations`/`interests`/`targetContext`/`decisionFactor`/`validationGoal`/`hypothesis` 등이 대응 컬럼이 없어 제출 시 그냥 버려짐.
**할 일**: `supabase/migrations/010_projects_extra_data.sql`로 `projects.extra_data JSONB` 추가 → `submitProject.ts`에서 이 필드들을 통째로 넣기 → 크리에이터 프로젝트 상세/리포트 프롬프트에서 필요할 때 참조

## 3그룹 — 운영/정합성 (급하지 않음, 방치하면 나중에 빚)

### 3-1. PSF 질문 매칭을 텍스트 대신 ID/타입 기반으로
**지금 상태**: `lib/ai/generateReport.ts`가 `PSF_STANDARD_QUESTIONS`의 정확한 문구를 텍스트로 매칭해서 서브스코어를 계산함. 나중에 누군가 그 문구를 1글자만 고쳐도 매칭이 조용히 깨지고 서브스코어가 전부 null이 된다.
**할 일**: `review_questions`에 `question_key`(예: `'psf_1'`, `'psf_3'`, `'sean_ellis'`) 컬럼을 추가해서 문구가 아니라 이 키로 매칭

### 3-2. `question_templates` vs 하드코딩 질문 상수 이중 관리 정리
**지금 상태**: 마법사가 보여주는 질문 문구는 `types.ts`의 상수, DB `question_templates`는 별개로 존재. 두 개가 다른 방향으로 수정될 위험.
**할 일**: 둘 중 하나로 통일 (권장: `question_templates`를 단일 소스로 하고 마법사가 이걸 조회하도록 변경 — 이미 `lib/api/questionTemplates.ts`에 조회 함수가 있어 재사용 가능)

### 3-3. 기한(deadline) 초과 처리
**지금 상태**: `projects.deadline`은 저장만 되고, 넘겨도 아무 일도 안 일어남 (자동 마감/알림 없음).
**할 일**: 매일 도는 Supabase Edge Function(cron)에서 `deadline < now() AND status='active' AND completed_count < target_count`인 프로젝트를 찾아 관리자에게 알리거나 자동으로 부분 완료 처리

### 3-4. 구 라우트 정리
**지금 상태**: `app/projects/[id]/report|review|distribution`(구버전)이 `app/builder/reports/[id]`, `app/evaluator/review/[id]`(현재 쓰는 버전)와 동시에 존재. 구버전엔 동작 안 하는 죽은 버튼도 있음.
**할 일**: 사용자가 "나중에 따로 알려주겠다"고 한 항목 — **착수 전 재확인 필요**
