## 이번에 실제로 고친 것

- `lib/distribution/execute.ts`: 브라우저 클라이언트(`lib/supabase/client`) → 서비스 롤 클라이언트(`lib/supabase/admin`)로 교체. RLS(009)로 인해 `reviewer_profiles`를 다른 리뷰어 것까지 못 읽어서 정산이 항상 깨지던 문제 해결
- `app/api/projects/[id]/distribute/route.ts`: 로그인 확인(`auth.getUser()`)과 소유권 검증(`project.creator_id === user.id`)이 **아예 없었던** 것을 추가 — 지금까지는 로그인 여부와 무관하게, 심지어 남의 프로젝트라도 `project.status === 'reviewing'` 조건만 맞으면(현재 파이프라인상 이 상태 자체가 안 쓰여서 사실상 도달 불가능하긴 했지만) 배분을 실행할 수 있는 구조였음

## C-3(즉시 active)을 적용하지 않는 이유

이 지적 자체는 맞다 — 결제 없이 등록 즉시 노출된다. 하지만 **이전 세션에서 이미 사용자와 합의한 사항**이다: "관리자 승인 화면(`/admin/requests`)은 다 만들어두되, 기본값은 여전히 `active`로 유지 — 테스트 편의 우선, 나중에 한 줄만 바꾸면 즉시 게이트 전환 가능." 이 합의를 뒤집는 변경이라 임의로 적용하지 않았다. 필요하면 말씀해주시면 바로 전환 가능하다.

## 남은 항목 진행 계획 (우선순위순)

### 다음 라운드 1순위
1. **H-1** 계좌 등록 페이지(`/reviewer/account-setup`)를 리뷰어 대시보드/지갑 화면에 링크 연결
2. **C-1 보강** — 완료 즉시 리포트가 생성되도록, 리뷰 제출 서버 처리를 하나의 API 라우트로 묶어서(`review_answers` insert + `project_matches` 갱신 + RPC + 완료 체크 + 리포트 생성까지 서버 트랜잭션 성격으로) 처리. 지금처럼 브라우저에서 여러 번 왕복하지 않게.
3. **M-3** `generateAndSaveReport()` 완료 후 `projects.status = 'completed'`로 갱신 (영향 적지만 간단해서 같이 처리 권장)

### 2순위
4. **H-4/H-5** 리뷰어 프로필 편집 화면 + 매칭 자진 취소 UI
5. **M-1** PSF 질문 매칭을 텍스트 대신 `question_type`/`question_templates` ID 기반으로 전환
6. **H-2** 저장 안 되는 마법사 필드들을 `projects.extra_data JSONB`에 보존

### 3순위 (급하지 않음)
7. **M-5** `deadline` 초과 시 처리 (Edge Function Cron 또는 관리자 수동 마감)
8. **H-3** 구 라우트(`app/projects/[id]/*`) 정리 — 사용자가 "나중에 별도로 알려주겠다"고 한 항목, 재확인 필요
