## 1. 프로젝트 개요

FindFit은 **PSF/PMF 검증 플랫폼**입니다. 크리에이터(창업자)가 제품/서비스를 등록하면, 리뷰어(평가단)가 신청 → 승인 → 제품 체험 → 설문 리뷰 작성 과정을 거쳐, 그 응답이 집계되어 크리에이터에게 AI 분석 리포트로 전달됩니다.

- **크리에이터 흐름**: 프로젝트 등록(마법사) → 지원자 승인/거절 → 목표 인원 달성 시 리포트 자동 생성 → 리포트 열람
- **리뷰어 흐름**: 피드에서 프로젝트 발견 → 지원 → 승인 대기 → (배송형이면 수령확인) → 리뷰 작성 → 완료/정산

## 2. 기술 스택 & 실행 방법

- **프레임워크**: Next.js 16 (App Router, Turbopack)
- **DB/Auth/Storage**: Supabase (Postgres + Auth + Edge Functions)
- **AI**: Google Gemini(`lib/ai/gemini.ts`) — 리포트 생성용. Claude(`lib/ai/claude.ts`)도 코드는 있지만 실제로는 안 쓰임(6번 섹션 참고)
- **이메일**: Resend (`RESEND_API_KEY`)
- **스타일**: Tailwind CSS, 커스텀 컬러(#F77019 크리에이터 오렌지 / #1565C0 리뷰어 블루)

```bash
npm run dev        # 로컬 개발 서버 (http://localhost:3000)
npx tsc --noEmit    # 타입체크
npx next build      # 프로덕션 빌드 검증
```

Supabase 마이그레이션은 CLI 연동이 안 돼 있어서, **Supabase 대시보드 → SQL Editor에서 `supabase/migrations/*.sql`을 번호 순서대로 직접 실행**하는 방식으로 반영해왔습니다. (9번 섹션에 최신 미적용 마이그레이션 안내)

## 3. 이번 세션에서 완료한 작업

작업 순서대로, 각 항목이 실제로 어떤 문제를 해결했는지 함께 정리합니다.

### 3-1. 크리에이터가 지원자를 직접 승인/거절
- **문제**: 리뷰어가 프로젝트에 지원해도, 승인 권한이 `/admin/applications`(관리자 전용, `ADMIN_SECRET_KEY` 쿠키 필요)에만 있어서 크리에이터 본인은 승인할 방법이 없었음. 그래서 리뷰어는 지원 후 영원히 "대기" 상태에 머무름.
- **해결**: `app/api/builder/matches/[matchId]/accept|reject/route.ts` 신규 — 세션으로 크리에이터 신원 확인 후, 서비스 롤 클라이언트로 소유권 검증(해당 프로젝트의 `creator_id`와 일치하는지) 후 `project_matches.status`를 갱신. `components/builder/ProjectDetailPage.tsx`에 "지원 승인 대기" 섹션 + 수락/거절 버튼 추가.

### 3-2. 리커트 문항 렌더링 버그
- **문제**: 크리에이터가 마법사에서 만든 커스텀 리커트(척도) 문항이 DB에 `question_type: 'likert'`로 저장되는데, 리뷰 작성 화면은 `'likert_5'`만 인식해서 **1~5점 버튼 자체가 렌더링되지 않고 빈 카드로만 표시**되던 버그. `'likert'` 타입도 동일하게 처리하도록 수정.

### 3-3. 리뷰어 대시보드 실데이터 연동
- **문제**: `components/evaluator/ReviewerDashboard.tsx`가 100% 하드코딩 더미 데이터(가짜 프로젝트 3개, 가짜 포인트 34,500P 등)였고, "의뢰 둘러보기"(`/evaluator/available`) 메뉴도 좌측 네비에 아예 없어서 우연히만 도달 가능했음.
- **해결**: 실제 Supabase 조회(정산 현황·등급·참여 가능 의뢰·내 참여 현황)로 전면 교체, 네비게이션에 "의뢰 둘러보기" 추가. (※ 이후 3-8에서 이 대시보드 자체가 카드형 단일화면으로 다시 한번 크게 바뀝니다.)

### 3-4. 크리에이터 대시보드 실데이터 연동 + 기타 UX 수정 (한 세트)
- `components/builder/CreatorDashboard.tsx`도 마찬가지로 100% 더미였던 것을 실제 Supabase 조회로 교체(진행 프로젝트/피드백/드래프트/완료 리포트/Fit Credit/최근 활동/모집 알림)
- **회귀 버그**: 위 작업 중 "프로젝트 0개면 위젯 전체를 등록 버튼 하나로 통째로 교체"하는 코드가 있어서, 실제 프로젝트가 적은 계정에서는 대시보드가 거의 텅 비어 보이는 문제가 생겼음 → 항상 위젯 레이아웃을 유지하고 개별 카드에서 빈 상태만 표시하도록 재수정.

### 3-5. 프로젝트 등록 마법사 개선 (접근방식 재배치 + 인원 제한 완화)

- 관리자 패널에 `/admin/stats` 신규 — 최근 7일 내 실제 활동(등록/지원/제출) 기준 크리에이터·리뷰어 활동 유저 수를 역할별로 분리 표시. (지금은 관리자 전용, 추후 사용자 공개 통계로 확장 예정이라고 사용자가 언급함)

### 3-7. `reviewer_profiles` row 없을 때 406 에러
- **문제**: `role-select`에서 `reviewer_profiles`를 upsert하는 로직이 생기기 전에 만들어진 계정은 해당 row가 아예 없는데, `/evaluator/wallet`·`/evaluator/profile`·`/api/projects/feed`가 `.single()`로 조회해서 0건일 때 PostgREST가 406을 던짐(콘솔에 계속 에러 로그).
- **해결**: `.maybeSingle()`로 전환 + row 없으면 그 자리에서 기본값으로 자가치유(upsert).

### 3-8. 리커트 문항 UX 개선 (2건)
1. **기준 라벨 상시 노출**: 1~5 버튼에 호버 전까지 아무 기준도 안 보여서 "1이 부정인지 5가 부정인지" 헷갈린다는 피드백 → 버튼 아래 "1 · 전혀 아니다 / 5 · 매우 그렇다"를 항상 보이게 표시. 라벨 문구도 낮음/높음 → 전혀 아니다~매우 그렇다(동의 정도)로 교체.
2. **1점(전혀 아니다) 선택 시 이유 자동 질문**: 크리에이터가 별도 후속 질문을 만들지 않아도, 리뷰어가 리커트에서 1점을 고르면 "왜 그렇게 느끼셨나요?" 입력칸이 자동으로 나타남. 이유 질문용 `review_questions` row가 따로 없어서, 제출 시 이유를 기존 답변 텍스트에 붙여(`"1 (이유: ...)"`) 리포트 생성 로직 변경 없이 그대로 반영되게 처리. **지금은 선택 입력**(안 적어도 제출 가능) — 필수로 바꿔달라는 요청은 아직 없었음.

### 3-9. 질문 등록 개선 + 리뷰 제출 검증 강화 (마이그레이션 020 포함)
- **객관식 복수선택**: `review_questions.allow_multiple BOOLEAN` 컬럼 추가(마이그레이션 020). 질문 작성 단계(`QuestionBuilder.tsx`)에서 객관식 문항에 "복수 선택 허용" 체크박스 추가 → 체크 시 리뷰어 화면에서 체크박스형 다중 선택으로 렌더(`ProjectCardExpandable.tsx`). 답변은 `"옵션1, 옵션2"` 콤마 조인 문자열로 저장(스키마·리포트 로직 추가 변경 없이 재사용).
- **미답변 스크롤**: 리뷰 제출 시 답 안 한 문항이 있으면 그 문항으로 자동 스크롤 + 2초간 빨간 테두리 강조.
- **성의없는 답변 차단**: 주관식 답변이 5자 미만이거나 `"..."`, 기호/공백만 있으면 제출을 막고 해당 문항으로 스크롤(`CARELESS_ANSWER_PATTERN` 정규식 + 최소 길이 체크, `ProjectCardExpandable.tsx`).

### 3-10. AI 리포트 생성 관련 확인 사항 (버그 아님, 설정 미비)
사용자가 "클로드 키가 없는데 리포트가 어떻게 생성되냐"고 질문해서 조사한 결과:
- 실제로 쓰이는 리포트 생성 코드(`lib/ai/generateReport.ts`)는 Claude가 아니라 **Gemini만** 호출하도록 하드코딩되어 있음.
- 그런데 `.env.local`의 `GEMINI_API_KEY`가 비어있어서, 지금 나오는 모든 리포트는 실제 AI 응답이 아니라 **완전 고정된 mock 데이터**(`lib/ai/gemini.ts`의 `getMockResponse`)임.
- 리포트 페이지의 "고급 분석 잠금 해제 (9,900원)" 버튼은 **onClick 핸들러 자체가 없음** — 결제 로직도 Claude 호출도 전혀 없는 순수 정적 목업(블러 처리된 미리보기 UI만 존재). Claude 키 유무와 무관하게 원래 아무 동작 안 함.
- `lib/ai/index.ts`에 크리에이터 레벨(`builder`/`launcher`)별로 Claude/Gemini를 분기하는 `generateReport()` 함수가 따로 있으나, **어디서도 호출되지 않는 죽은 코드**. 실제 파이프라인은 이 함수를 안 씀.

### 권한/RLS 핵심 규칙 (migration 009)
- `projects` 테이블 원본은 **소유자(크리에이터)만** 전체 접근. 리뷰어는 `projects_public` 뷰(creator_id 제외)로만 조회.
- `project_matches` 원본은 **리뷰어 본인 row만** 접근. 크리에이터는 `project_matches_for_creator` 뷰(reviewer_id/이메일 제외)로만 조회.
- 크리에이터가 지원자 승인/거절/배송상태 변경 등 리뷰어 소유 row를 대신 써야 할 때는 전부 **서버 API에서 소유권 수동 검증 + 서비스 롤 클라이언트로 우회**하는 패턴을 씀(`app/api/builder/matches/[matchId]/*`). 새 기능 추가 시 이 패턴을 따르면 됨.

## 9. 자주 겪었던 로컬 개발 이슈 & 해결법

이번 세션 내내 반복됐던 문제라 다음 작업자도 겪을 가능성이 높아 기록해둡니다.

**증상**: `npx next build` 등으로 빌드 검증 후 `.next` 폴더를 정리(`rm -rf .next`)했는데, 기존에 떠 있던 `npm run dev` 프로세스를 재시작하지 않으면 → 그 좀비 프로세스가 이미 삭제된 빌드 산출물을 계속 찾다가 `GET /evaluator/dashboard 500 (Internal Server Error)` 같은 에러가 남.

**해결법**:
```bash
# 1) 3000번 포트 점유 프로세스 확인 후 종료 (PowerShell)
Get-NetTCPConnection -LocalPort 3000 -State Listen | Select-Object -ExpandProperty OwningProcess | ForEach-Object { Stop-Process -Id $_ -Force }

# 2) 새로 시작
npm run dev
```
`.next`를 지웠다면 **반드시 dev 서버를 재시작**해야 합니다. (매번 까먹기 쉬우니 주의)
